modulos

Los modulos son como contenederos que agrupan componentes realacionados como:
controladores, servicios, proveedores, etc bajo el mismo contexto.
ayuda al encansulamiento porque cada modulo es indepediente y organiza el codigo, permite
la reutilizacion porque cada modulo puede exportar cierto servicios y componente evitando
la duplicacion logica del codigo.
permite la inyeccion de dependencia, permite compartir entre modulos.
Todo proyecto de nest tiene un AppModule (modulo raiz) que sirve como punto
de entrada y donde se importa los demas modulos:
Ejemplo basico de AppModule 
      
        import { Module } from '@nestjs/common';
        import { UsersController } from './users.controller';
        import { UsersService } from './users.service';
        
        @Module({
          controllers: [UsersController],
          providers: [UsersService],
          exports: [UsersService], // permite que otros módulos usen este servicio
        })
        export class UsersModule {}
        
        En este ejemplo:
        
        UsersController maneja las rutas relacionadas con usuarios.
        UsersService contiene la lógica de negocio.
        UsersModule agrupa todo y puede compartir el servicio con otros módulos.
        
        👉 En resumen: los módulos en NestJS son la columna vertebral de la arquitectura, permiten organizar, encapsular y escalar aplicaciones de manera limpia y modular.
Por convencion se recomienda que cada modulo exista en un carpeta idependiente en minuscula y en plural osea:
a excepcion de app que no debe estar en carpeta si en la parte externa de la aplicacion. 
      
        users/
        orders/
        products/
        invoices/
        roles/
        permissions/
exception en singular por ser modulos conceptuales y no representan CRUD de la aplicacion 
      
        auth/
        config/

        health/

service

dentro del service creamos la funcione que ejecuta toda la logica, los controler, simplemente
muestra la ruta que debemos retornar, pero la logica debe estar en los service. 
      
        import { Injectable } from "@nestjs/common";
        
        @Injectable()
        export class TaskService{
        
        }
podemos notar que la clase es parecida a la de modulo y a la de controladores, pero en este caso,
se llama injectable, y en el decorador tampoco hay que inicializar un objeto vacio.
ahora dentro de la clase crearemos la funcion que me va a retorna un array : 
    
      import { Injectable } from "@nestjs/common";
      
      @Injectable()
      export class TaskService{
      
          getTasks(){
              return ["task1","task2","tasks3"]
          }
      }
ahora nuestro controller debe llamar a nuestro service y necesitamos crear un contructor el codigo es el
siguiente: 
    
      import { Controller, Get } from "@nestjs/common";
      import { TasksService } from "./tasks.service";
      
      @Controller({})
      export class TasksController {
          tasksService: TasksService;
      
          constructor(tasksService: TasksService) {
              this.tasksService = tasksService;
          }
          @Get('/tasks')
          getAllTasks() {
              return this.tasksService.getTasks();
          }
      }
por ultimo para nuestro controlador sepa que va a utilizar un service lo debemos expecificar en el modulo: 
    
      import { Module } from "@nestjs/common";
      import { TaskController } from "./tasks.controller";
      import { TasksService } from "./tasks.service";
      
      
      @Module({controllers:[TaskController], providers:[TasksService]})
      export class TaskModule{
      
      }
como podemos notar nuestro codigo de controller. esta algo verboso se puede mejorar así: 
    
      import { Controller, Get } from "@nestjs/common";
      import { TasksService } from "./tasks.service";
      
      @Controller('tasks')
      export class TasksController {
      
          constructor(private readonly tasksService: TasksService) {

          }
          @Get()
          getAllTasks() {
              return this.tasksService.getTasks();
          }
      }
Nota de mejoria del codigo
el controlador no debe tener un objeto vacio, sino la ruta principal del controlador tasks, esta ruta hay que quitarla,
del get() porque tendriamos que escribirla por cada funcion que hagamos.
con respecto al codigo anterior se cambio dentro del contructor una variable local quedando así:
private readonly tasksService: TasksService y no hay inicializar nada ni que recibir el paso de una variable a otra local. este concepto se llama injection de dependencia simplificado

response

En NestJS, Response normalmente se refiere al objeto de respuesta HTTP que proviene de Express (o Fastify, según el adaptador que uses).
NestJS está construido sobre Express (por defecto) o Fastify.
Cuando inyectas @Res() en un controlador, lo que recibes es el objeto Response de Express/Fastify.
Este objeto te permite controlar manualmente la respuesta HTTP: enviar JSON, redirigir, establecer cabeceras, cookies, etc.
un ejemplo con express seria: 
      
        import { Controller, Get, Res } from '@nestjs/common';
        import { Response } from 'express';
        
        @Controller('tasks')
        export class TasksController {
          @Get()
          getTasks(@Res() res: Response) {
            const tasks = [{ id: 1, name: 'Test task' }];
            res.status(200).json(tasks);
          }
        }
pero segun las mejores practicas debe ser y NestJS se encarga de la conversion de json : 
      
        @Get()
        getTasks() {
          return [{ id: 1, name: 'Test task' }];
        }
Para JSON y respuestas normales, mejor retornar directamente y dejar que NestJS maneje todo.
Para descargas, redirecciones o streams, usa @Res() porque necesitas control manual.
ejemplo de descaga 
      
        import { Controller, Get, Res } from '@nestjs/common';
        import { Response } from 'express';
        import { join } from 'path';
        
        @Controller('files')
        export class FilesController {
          @Get('download')
          downloadFile(@Res() res: Response) {
            const filePath = join(__dirname, '..', 'assets', 'example.pdf');
            res.download(filePath, 'mi-archivo.pdf'); 
          }
        }

Reques Body

En NestJS, el request body es el contenido que envía el cliente dentro de una petición HTTP
(normalmente en métodos POST, PUT o PATCH). Es la parte del mensaje donde se transmiten los datos que
quieres procesar en tu servidor, por ejemplo un objeto JSON con información de una tarea, usuario, producto, etc.

codigo task.controller.ts 
      
        import { Controller, Get, Post, Put, Delete, Patch, Body } from "@nestjs/common";
        import { TasksService } from "./tasks.service";
        
        @Controller('tasks')
        export class TasksController {
        
            constructor(private readonly tasksService: TasksService) {
        
            }
        
            @Get()
            getAllTasks() {
                return this.tasksService.getTasks();
            }
        
            @Post()
            createTasks(@Body() task: any) {
                return this.tasksService.createTask(task);
            }
        
            @Put()
            updateTasks() {
                return this.tasksService.updateTask();
            }
        
            @Delete()
            deleteTasks() {
                return this.tasksService.deleteTask();
            }
        
            @Patch()
            updateTasksState() {
                return this.tasksService.updateTaskState();
            }
        }
utilizamos el decorador @Body(), el cual es importado desde @nestjs/common, y lo pasamo como parametro al servicio

validaciones

En NestJS las validaciones se implementan principalmente con ValidationPipe y las librerías class-validator y class-transformer.
Se definen reglas en los DTOs con decoradores, y el pipe se encarga de verificar automáticamente los datos entrantes.
Para trabajar con las validaciones inicialmente hay que instalar las librerias asi: 
      
        npm install class-validator class-transformer
codigo de validacion dentro del dto de crear tarea: 
      
        import { IsBoolean, IsNotEmpty, IsNumber, IsString, MinLength } from "class-validator"
        
        export class CreateTaskDto {
            @IsNumber()
            id: number
        
            @IsString()
            @IsNotEmpty()
            @MinLength(10)
            description: string
        
            @IsBoolean()
            state: boolean
        }
para que la validacion se aplique de forma global hay que importar ValidationPipe dentro del main.ts 
      
        import { NestFactory } from '@nestjs/core';
        import { AppModule } from './app.module';
        import { ValidationPipe } from '@nestjs/common';
        
        async function bootstrap() {
          const app = await NestFactory.create(AppModule);
          app.useGlobalPipes(new ValidationPipe());
          await app.listen(process.env.PORT ?? 3000);
        }
        bootstrap();

Pipeline

Cada petición HTTP pasa por un pipeline de ejecución.
Este pipeline está compuesto por middlewares, guards, interceptors, pipes y filters.
Los pipes son una parte clave: sirven para transformar y validar los datos que entran a tu aplicación.

Por ejemplo si creamos una funcion token que recibe un numero y retorno otro sumado 10, quedaria así; 
      
        import { Controller, Get, Post, Put, Delete, Patch, Body, Query, Param, ParseIntPipe } from "@nestjs/common";


        @Get('ticket/:num')
        getToken(@Param("num", ParseIntPipe) num: number){
            console.log(num);
            return num+10;
    
        }
y para probarlo escribimos: 
      
        http://localhost:3000/tasks/ticket/2

Guards

un Guard es una clase que decide si una petición puede acceder a un controlador o ruta.
Se usa principalmente para autenticación y autorización, verificando permisos, roles o condiciones antes de ejecutar el método del controlador

Ejemplo basico de Guards 
      
        import { Injectable, CanActivate, ExecutionContext } from '@nestjs/common';

convenciones en base de datos

convenciones para nombre de tablas y columnas 
      
        Nombres de tablas y columnas:
        
        Usa snake_case (ej: user_account, order_item).
        Siempre en singular para tablas que representan entidades (user, product).
        Evita abreviaturas poco claras.
        
        Esquemas:
        
        Usa esquemas para separar dominios (ej: auth.user, sales.order).
        Mantén un esquema public solo para pruebas o datos comunes.
        Define permisos por esquema para mayor seguridad.
        
        Llaves primarias y foráneas:
        
        Usa id como PK con tipo BIGSERIAL o UUID.
        Nombra las FK con el patrón _id (ej: user_id).
        
        Índices:
        
        Crea índices en columnas usadas en WHERE, JOIN y ORDER BY.
        Usa índices únicos para garantizar integridad.
        para los nombres se recomienda idx__
        para FK constraints se recomienda fk__
        unique se recomienda uq__
        
        Migraciones:
        Siempre versiona cambios en la base de datos.
        Documenta decisiones de diseño en cada migración.

SGBD Sistema de Gestion de Base de Datos

Primero que todo hay que entender que Postgres en un sistemas de gestion multi proceso no multihilo.
esto quiere de decir que el servidor esta escuchando y varios procesos estan corriendo solicitnao informacion,
como una pagina hecha con Next, o un administrador de DB como DBeaver, etc.
por cada conexion se crea un servicio independiente para el sistema operativo, de tal forma que no interfiere unos con los otros, en cambio Mysql
maneja una sola conexion con multiples hilos (multi-thread).
ventajas de postgres, si un proceso falla no interfiere con los demas, en cambio en mysql si la conexion falla todos los procesos fallan,
es por eso que postgres nececita mas memoria que mysql. 

      
        💡 Consejo de DBA: en aplicaciones modernas, para evitar saturar el servidor con demasiados procesos, 
        se recomienda usar un connection pooler como PgBouncer o Pgpool-II, que gestionan las conexiones de manera más eficiente.
PostgreSQL es un motor cliente-servidor con arquitectura multi-proceso (no multi-thread). 
      ┌───────────────────────┬──────────────────────────────────────┐
      | Componente            | Función                              |
      | --------------------- | ------------------------------------ |
      | Postmaster            | Proceso padre, gestiona conexiones   |
      | Backend process       | Un proceso por conexión              |
      | Shared Buffers        | Caché de datos en memoria            |
      | WAL (Write-Ahead Log) | Garantiza durabilidad                |
      | Autovacuum            | Limpieza y estadísticas              |
      | System Catalogs       | Metadatos (pg_class, pg_index, etc.) |
      └───────────────────────┴──────────────────────────────────────┘

      📌 Regla DBA:
      
      Si no entiendes WAL y Autovacuum, tarde o temprano tendrás problemas de rendimiento.

PostgreSQL es 100% ACID : 
      
      🔑 ¿Qué es ACID?
      ACID son las siglas de cuatro propiedades fundamentales que garantizan la fiabilidad de las transacciones en una base de datos:
      
      Atomicity (Atomicidad)
      Una transacción se ejecuta completa o no se ejecuta en absoluto.
      Ejemplo: si insertas un pedido y sus items, o se guardan todos o ninguno.
      PostgreSQL asegura esto con su sistema de transacciones y rollback.
      
      Consistency (Consistencia)
      Después de una transacción, la base de datos siempre pasa de un estado válido a otro estado válido.
      Las reglas de integridad (constraints, foreign keys, triggers) garantizan que no queden datos corruptos.
      Ejemplo: no puedes tener un order_item que apunte a un order inexistente.
      
      Isolation (Aislamiento)
      Las transacciones concurrentes no interfieren entre sí.
      PostgreSQL usa MVCC (Multi-Version Concurrency Control) para que cada transacción vea una “foto” consistente de los datos, sin bloquearse innecesariamente.
      Ejemplo: dos usuarios pueden comprar productos al mismo tiempo sin pisarse los datos.
      
      Durability (Durabilidad)
      Una vez confirmada (commit) la transacción, los datos quedan grabados de forma permanente, incluso si hay un fallo del sistema.
      PostgreSQL usa WAL (Write-Ahead Logging) para asegurar que los cambios se registren primero en un log antes de aplicarse, lo que permite recuperación tras fallos.        

      Ejemplo practico:
      
      BEGIN;
      
      INSERT INTO orders (...) VALUES (...);
      INSERT INTO order_items (...) VALUES (...);
      
      COMMIT;
      -- o
      ROLLBACK;

Install Postgresql in linux

Paso 1 — Preparar sistema
        
          sudo apt update && sudo apt upgrade -y
          sudo apt install wget gnupg2 lsb-release -y
Son los paquetes que se van a instalar:

wget → herramienta para descargar archivos desde internet por terminal.
gnupg2 → herramientas de cifrado y verificación de firmas (muy usado para validar repositorios).
lsb-release → muestra información de la distribución Linux (nombre, versión, etc.).

Paso 2 — Importar llave 
      
        curl -fsSL https://www.postgresql.org/media/keys/ACCC4CF8.asc \
        | gpg --dearmor \
        | sudo tee /usr/share/keyrings/postgresql.gpg > /dev/null
explicacion del comando: 
      

        curl -fsSL URL
    
        Función: descarga la clave pública
    
        Internamente:
        abre conexión HTTPS
        valida certificado TLS
        descarga el archivo .asc
    
        lo envía a stdout (flujo estándar)
    
        👉 No guarda archivo en disco
        👉 Lo manda directo al pipeline
    
        Pipe | 
        El símbolo pipe hace:
        No se crean archivos temporales.
        Se usan buffers en memoria del kernel.
    
        gpg --dearmor
    
        Función: convertir clave ASCII → binario GPG
    
        Explicación técnica:
        GPG recibe texto ASCII
        lo parsea
        lo transforma a formato binario OpenPGP

        sudo tee /usr/share/keyrings/postgresql.gpg
        
        Función: guardar archivo con permisos root
        Detalles importantes:
        
        tee escribe en archivo y stdout
        sudo es necesario porque /usr/share/keyrings/ es root-only
        el archivo se crea con permisos seguros

        > /dev/null
        
        Esto redirige la salida de tee para que no muestre basura binaria en pantalla.
        Sin esto verías caracteres ilegibles.
Nota:
este comando ya esta depreciado y era el que me mostraba la IA : 
      
         wget -qO - https://www.postgresql.org/media/keys/ACCC4CF8.asc | sudo apt-key add -
explicacion del comando : 
      
        wget -qO - https://www.postgresql.org/media/keys/ACCC4CF8.asc 

  
        wget → herramienta para descargar archivos desde internet. 

        -q → modo silencioso (quiet), no muestra mensajes. 

        -O - → indica que el archivo descargado se envíe a la salida estándar (pantalla) en vez de guardarse en un archivo. 

  
        La URL → es la clave pública oficial del repositorio. 

        👉 Resultado: descarga la clave y la envía directamente al siguiente comando. 


  
        | El pipe pasa la salida de wget como entrada al siguiente comando. 


        sudo apt-key add - 

  
        apt-key → herramienta para gestionar claves de confianza de repositorios APT. 

        add → agrega una nueva clave. 

        - → significa “leer desde entrada estándar” (lo que viene del pipe). 

        sudo → necesario porque estás modificando claves del sistema. 

  
        👉 Resultado: se agrega la clave al sistema para que APT pueda verificar que los paquetes del repositorio son
        auténticos.
Paso 3 — Agregar repositorio usando la clave 
      
         echo "deb [signed-by=/usr/share/keyrings/postgresql.gpg] http://apt.postgresql.org/pub/repos/apt $(lsb_release -cs)-pgdg main" \
        | sudo tee /etc/apt/sources.list.d/pgdg.list
explicancion : 
      
        echo "deb ... main"

        echo imprime texto.
        Ese texto es exactamente el formato que APT usa para definir repositorios.

        $(lsb_release -cs)

        Esto es expansión de comando.

        El sistema ejecuta:
        lsb_release -cs

        y devuelve el nombre de tu distribución.
        Por ejemplo Ubuntu 22.04, Debian 12, etc

        👉 Esto asegura que el repo coincide con tu sistema operativo.

        deb [signed-by=...] URL distro main

        Esta es la línea de configuración APT.

        Significado:

        Parte	Significado
        deb	repositorio binario
        [signed-by=...]	clave GPG específica
        URL	servidor repo
        distro-pgdg	versión distro
        main	sección paquetes

        🔐 signed-by= (IMPORTANTE)
        [signed-by=/usr/share/keyrings/postgresql.gpg]

        Esto le dice a APT:

        Solo confía en paquetes firmados con esta clave.

        Pipe |

        Pasa la salida de echo al siguiente programa.
        echo → tee

        sudo tee /etc/apt/sources.list.d/pgdg.list

        tee escribe el contenido en archivo.

        ¿Por qué sudo tee y no sudo echo?

        Porque redirecciones (>) no funcionan con sudo directo.
        ✔ correcto:

        echo texto | sudo tee archivo

        ❌ incorrecto:

        sudo echo texto > archivo

        Ruta del archivo
        /etc/apt/sources.list.d/

        Esta carpeta es donde se almacenan repositorios adicionales.

        APT carga automáticamente todos los .list de ahí.   

        🏆 Por qué este método es el estándar moderno

        Este patrón:

        echo | sudo tee

        2️⃣ Configurar PostgreSQL para conexiones remotas

        Hay 2 archivos clave:

        postgresql.conf	Configuración general
        pg_hba.conf	Control de acceso

        Editar postgresql.conf

        Ubicación común:
        /etc/postgresql/18/main/postgresql.conf

        Buscar:
        listen_addresses =

        Cambiar a:
        listen_addresses = '*'

        Significa:
        PostgreSQL acepta conexiones desde cualquier IP

        Buena práctica en producción:
        listen_addresses = '192.168.1.50'

        (solo tu IP interna)

        Editar pg_hba.conf

        Archivo:
        /etc/postgresql/18/main/pg_hba.conf

        Agregar al final:
        host    all     all     0.0.0.0/0     md5

        | Campo     | Valor                      |
        | --------- | -------------------------- |
        | host      | conexiones remotas         |
        | all       | todas las bases            |
        | all       | todos los usuarios         |
        | 0.0.0.0/0 | cualquier IP               |
        | md5       | autenticación con password |

        🔐 Más seguro:

        host all all 192.168.1.0/24 md5
        (solo red local)
